# Plan: Generic Wave Function Collapse Engine for Excalibur.js + Vite

**TL;DR:** Implement a fully generic WFC algorithm in `src/wfc/` that reads tile types and neighbor constraints from JSON, performs entropy-driven collapse with constraint propagation and backtracking, and exposes an adapter to map results into Excalibur's TileMap system. The engine supports asymmetric neighbor constraints (different allowed neighbors per side) and optional symmetry modes.

### Symmetry Modes Explained

- `"none"` — constraints are applied as-is (asymmetric)
- `"horizontal"` — if tile A can be to the right of B, then B can be to the left of A
- `"vertical"` — if tile A can be below B, then B can be above A
- `"both"` — apply both horizontal and vertical symmetry
- `"rotational"` — 90° rotational symmetry: top↔right↔bottom↔left cycle

  bottom: Set<string>;

export interface TileTypeInternal {
  id: string;
  name: string;
  spriteRef?: string;
  weight: number;

/** Result of running the WFC algorithm. */

  complete: boolean;
  /** Number of backtracking steps performed */
  backtrackCount: number;
}

### File 2: `src/wfc/engine.ts` (NEW — main WFC engine class)

**Purpose:** Core Wave Function Collapse algorithm implementation.

#### Class: `WaveFunctionCollapse`

```ts
export class WaveFunctionCollapse {
  private gridWidth: number;
  private gridHeight: number;
  private tileTypes: Map<string, TileTypeInternal>;
  private grid: GridCell[][];
  private symmetryMode: SymmetryMode;
  private backtrackCount: number = 0;

  constructor(config: WFCConfig);

  /** Run the full algorithm. Returns a solution or throws on irrecoverable contradiction. */
  solve(): WFCSolution;

  // --- Internal methods ---
  private initializeGrid(): void;
  private calculateEntropy(cell: GridCell): number;
  private findLowestEntropyCell(): { x: number; y: number } | null;
  private collapseCell(x: number, y: number): boolean;
  private propagateConstraints(x: number, y: number): boolean;
  private propagateToNeighbor(cx: number, cy: number, nx: number, ny: number, side: NeighborSide): void;
  private backtrack(): boolean;
  private takeSnapshot(): GridSnapshot;
  private restoreSnapshot(snapshot: GridSnapshot): void;

  // --- Symmetry helpers ---
  private applySymmetryToConstraints(tileConfig: TileTypeConfig): TileConstraints;
}
```

#### Algorithm Flow (in `solve()`):

1. **Initialize** — create grid where every cell has all tile IDs as possibilities
2. **Loop** until complete or max iterations:
   a. Find the non-collapsed cell with lowest entropy (fewest remaining possibilities)
   b. If no such cell exists → return solution (complete = true)
   c. Collapse that cell: pick one possibility weighted by `weight`
   d. Propagate constraints to all neighbors recursively
   e. If propagation causes a contradiction (empty set on any cell):
      - Try backtracking (restore previous snapshot, re-collapse with different choice)
      - If no snapshots remain → throw error or return incomplete solution

#### Key Design Decisions:

- **Entropy** = `possibilities.size` — simple count-based entropy. Lower = more constrained.
- **Collapse selection**: weighted random based on tile `weight`. Use cumulative distribution for O(n) selection.
- **Propagation**: when cell (x,y) collapses to tile T, for each neighbor N:
  - Determine which side of N faces (x,y): e.g., if N is above (x,y), then N's "bottom" constraint must include T
  - Intersect N's possibilities with the allowed set from that side
- **Backtracking**: maintain a stack of snapshots taken before each collapse. On contradiction, pop and retry.

## 4. Edge Cases & Testing Considerations

### Edge Cases to Handle

1. **Empty neighbor lists**: If a tile has no allowed neighbors on a side, it can only appear at grid edges facing that direction
2. **Contradiction with no backtracking path**: The algorithm may reach an irrecoverable state. Return `complete: false` and log the issue rather than throwing
3. **Single-tile configs**: WFC should handle grids where all cells must be the same tile type
4. **Large grids**: Entropy calculation scans all cells each iteration — consider maintaining a priority queue for performance on large grids (>50x50)
5. **Symmetry conflicts**: If symmetry mode is set but constraints are already symmetric, no-op; if they conflict, log warning and apply the more restrictive constraint

### Testing Strategy

- **Unit tests** (if test framework exists): Test entropy calculation, collapse selection, propagation logic with small 3x3 grids
- **Manual verification**: Run WFC on sample config, output grid to console as ASCII art, verify visually that constraints are respected
- **Edge case testing**: 
  - Config with only one tile type → should produce uniform grid instantly
  - Config with contradictory constraints → should return incomplete solution gracefully
  - Large grid (100x100) → measure execution time

### Performance Notes

- For grids up to ~30x30, the naive O(n²) entropy scan is fine
- For larger grids, consider maintaining a sorted list of cells by entropy and updating it incrementally during propagation
- Backtracking can be expensive; add a `maxBacktracks` parameter (default 1000) to prevent runaway

## Decisions

- **No external dependencies**: The WFC engine is pure TypeScript with no third-party libraries. Only Excalibur types are imported in the integration file.
- **JSON-first design**: All configuration comes from JSON, making it easy to swap tile sets without recompiling
- **Backtracking over restart**: On contradiction, we backtrack to the last decision point rather than restarting the entire grid — this is more efficient for sparse contradictions
- **Weighted random collapse**: Tiles with higher `weight` are selected more frequently during collapse, allowing designers to bias generation toward certain patterns

## Further Considerations

1. **Should the engine support "don't care" neighbors?** Currently if a tile omits a neighbor side entirely, it defaults to "any tile allowed." An alternative is explicit `"*"` wildcard in JSON. Recommendation: keep implicit-any for simplicity; add wildcard only if requested.
2. **Should propagation be BFS or recursive?** Recursive is simpler and works well for small grids. For large grids, iterative BFS with a queue avoids stack overflow. Recommendation: start recursive, refactor to iterative if grid sizes exceed 50x50.
3. **Should the integration layer support animated tiles?** Excalibur SpriteSheet supports animation frames. The current design maps one spriteRef per tile ID. If animated tiles are needed, extend TileTypeConfig with an `animationFrames` field and update the adapter to handle it.
